# To PRD

把“想做什么”整理成“可以拆解和实现什么”。这个 skill 产出的 PRD 是工程输入，不是长篇产品文档。

## 核心原则

- MVP 优先：先定义最小可验证价值，不追求完整愿景。
- 只记录影响实现的信息：用户、问题、成功标准、范围、约束、关键决策。
- 使用项目领域语言：优先读取 `CONTEXT.md`；如有相关 ADR，尊重已确认决策。
- Deep Module 意识：描述功能时主动寻找可由小接口封装的能力域。
- 默认本地输出：生成 PRD 文本或写入本地文档；只有仓库上下文明确配置了 issue tracker 时，才发布到外部系统。

## 使用流程

### 1. 收集上下文

如果 `docs/generated/discovery.md` 存在，**先读它**：Discovery Brief 中已确认的事实（目标用户、问题、成功指标、替代方案、假设）直接作为 PRD 的输入，不重新收集。

在此基础上补充仓库事实：

- README、CONTEXT、ADR 和相关代码。
- 现有模块、接口、测试和 flow。

不要为了填模板采访用户。只有缺失信息会改变范围、接口或验收标准时，才提出一个关键问题，并给出推荐默认值。

### 2. 收敛 MVP 边界

如果来自 Discovery，以下字段已确认，直接携带，不重新讨论：

- 目标用户（来自 `Target users`）。
- 核心问题（来自 `Problem`）。
- MVP 成功指标（来自 `MVP success metric`）。

需要在 PRD 阶段补充的：

- 哪些能力在本轮范围内，哪些留到以后。
- 有哪些技术、时间、兼容性或组织约束。

如果出现高影响 trade-off，交给 `decision-forge` 收敛。

### 3. 梳理实现视角

概述将要构建或修改的主要 Module：

- 每个 Module 隐藏什么复杂度。
- 对外 Interface 大致承担什么语义。
- 哪些变化点需要被隔离。
- 哪些行为值得优先测试。

不要在 PRD 中写具体文件路径或代码片段，避免很快过时。

### 4. 生成 PRD

使用下面模板。保持简洁，能指导拆 issue 和实现即可。

```markdown
## 问题陈述

从目标用户视角描述当前问题，以及为什么值得解决。

## MVP 目标

描述本轮最小可验证价值。说明用户完成什么动作后，项目可以认为 MVP 成立。

## 目标用户

- 主要用户：
- 次要用户：

## 解决方案

描述用户可感知的解决方案，不堆实现细节。

## 用户故事

1. 作为一个 <actor>，我想要 <feature>，以便 <benefit>。

## 范围

### 范围内

- ...

### 范围外

- ...

## 实现决策

- 主要 Module：
- 关键 Interface：
- 数据或 API 契约：
- 架构约束：
- 已有 ADR 对齐：

## 测试决策

- 关键路径：
- 边界行为：
- 异常行为：
- 可复用测试先例：

## 风险与开放问题

- ...
```

## 发布规则

- 默认输出为本地 PRD 文档或对话中的 PRD 内容。
- 如果仓库明确配置了 GitHub、GitLab、Jira、Linear 或本地 markdown issue tracker，可以按其规则发布。
- 发布时应用仓库配置的 triage 标签；没有配置时不要臆造外部流程。

## 禁止行为

- 生成百科式长 PRD。
- 用未定义的同义词替代 CONTEXT 中已有术语。
- 把实现细节、文件路径和临时代码写进 PRD。
- 强制创建 issue tracker 条目。
- 把所有未来愿景塞进 MVP。
