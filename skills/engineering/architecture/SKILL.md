# 架构改进

这个 skill 用于发现架构摩擦，并提出能降低认知成本的改进候选。它不套用某一种固定架构风格，而是把 `methodology.md` 的统一架构方法论落到当前代码库。

目标不是“看起来更优雅”，而是让系统更容易理解、修改、测试和交给 AI 协作。

## 判断原则

按以下优先级评估所有建议：

| ID | 原则 | 审查问题 |
| --- | --- | --- |
| U1 | 降低复杂度优先 | 这个改动是否降低理解和修改系统的认知成本？ |
| U2 | Deep Module 优先 | 是否用小接口隐藏了足够多的实现复杂度？ |
| U3 | 接口最小化 | 调用者是否只需要知道必要能力和契约？ |
| U4 | 隐藏变化 | 变化是否被局部化，而不是扩散到多个调用方？ |
| U5 | 依赖抽象 | 高层逻辑是否依赖稳定语义，而不是具体技术实现？ |
| U6 | 组合优于继承 | 扩展是否避免破坏父类契约和结构耦合？ |
| U7 | 层级抽象一致 | 同一模块内是否混入了不同抽象层级？ |
| U8 | 模式服务于简化 | 设计模式是否真的降低复杂度，而不是增加间接性？ |

当原则冲突时，以 U1 为最终判断标准。

## 语言规范

架构建议统一使用 [LANGUAGE.md](LANGUAGE.md) 中的词汇：

- **Module**：有接口和实现的任何单元，可以是函数、类、包、feature slice 或 bean cluster。
- **Interface**：调用者必须知道的一切，包括类型、约束、不变量、错误模式、调用顺序和配置。
- **Depth**：接口带来的杠杆。Deep 表示小接口背后有大量能力；Shallow 表示接口几乎和实现一样复杂。
- **Seam**：可以替换行为而不原地修改的接口位置。
- **Adapter**：满足 seam 上接口的具体实现。
- **Leverage**：调用者从深模块得到的能力。
- **Locality**：维护者从深模块得到的局部修改能力。

不要把框架名当成架构结论。`Service`、`Repository`、`Controller`、`Handler`、`Client` 只有在创造 leverage 和 locality 时才有架构价值。

## 使用流程

### 1. 对齐上下文

先读取：

- `CONTEXT.md` 或相关上下文的领域词汇。
- `docs/adr/` 中与当前区域相关的决策。
- 代码布局、语言、框架、测试风格和依赖方式。

如果发现建议会冲突 ADR，必须显式标记；只有真实摩擦足够高时才建议重开决策。

### 2. 识别架构摩擦

重点寻找这些 signals：

- 修改一处需要改多处，出现 change amplification。
- 理解一个概念需要在许多小模块间跳转。
- 模块只是转发调用，接口复杂但实现很薄。
- 接口参数多、调用顺序隐含、错误模式分散。
- 领域逻辑和技术细节混在同一层。
- if/else 特例爆炸，业务规则散落在调用方。
- 单元测试需要大量 mock 内部协作者。
- 外部依赖、数据库、网络协议泄露到核心逻辑。
- 抽象只有一个真实 adapter，且没有测试或演进价值。

对可疑模块做 deletion test：

```text
删除这个 module 后，复杂度是消失了，还是重新散落到 N 个调用方？
```

如果复杂度只是消失，它可能是浅模块。如果复杂度会扩散，它可能正在创造价值。

### 3. 提出深化候选

先只提出候选，不直接设计新接口。每个候选包含：

- **位置**：涉及的文件、模块或调用链。
- **问题**：当前复杂度如何表现。
- **建议**：要加深、合并、隔离或移动什么。
- **收益**：用 leverage、locality、测试可达性说明价值。
- **代价**：短期改动成本、兼容风险或框架约定冲突。
- **方法论映射**：对应 U1-U8、Deep Module、接口契约或 smells。

推荐输出 3-5 个候选，按影响和风险排序。

### 4. 设计选中候选

用户选中候选后，再进入接口设计：

- 定义新的 Interface，包括输入、输出、不变量、错误模式和调用顺序。
- 判断依赖类型：in-process、local-substitutable、remote but owned、true external。
- 决定 seam 和 adapter 是否必要；一个 adapter 通常只是猜测，两个 adapter 才更可能是真 seam。
- 说明哪些复杂度被移入 Implementation，哪些信息不再暴露给调用方。
- 设计测试：测试必须穿过新 Interface 验证行为，不测试内部结构。

如需比较多个接口方案，按 [接口设计](INTERFACE-DESIGN.md) 执行。

### 5. CONTEXT 同步

接口设计完成后，检查是否有需要固化的知识：

- 新 Interface 引入的稳定领域术语 → 写入 `CONTEXT.md`。
- 模块边界或职责范围的确认 → 写入 `CONTEXT.md`。
- 有真实 trade-off 的架构决策 → 通过 `decision-forge` 创建 ADR。

惰性原则：只在真实需要固化时更新；不要为了"完整性"补充早已隐含的知识。

## 模式选择约束

只有在能降低认知成本时才引入模式：

| 问题 | 可考虑模式 |
| --- | --- |
| 接口太复杂，调用方负担高 | Facade |
| 外部依赖变化或协议泄露 | Adapter |
| 行为变体和 if/else 泛滥 | Strategy / Template Method |
| 对象构造步骤复杂 | Builder / Factory |
| 顺序处理且职责分离清晰 | Pipeline / Chain |
| 横切能力需要透明增强 | Proxy / Decorator |

禁止为了“符合某架构”而搬运模式。模式必须服务于简化。

## 禁止行为

- 不读 CONTEXT / ADR 就给架构建议。
- 默认推荐 Clean Architecture、DDD、Hexagonal、微服务或任意固定模板。
- 把纯粹命名或目录调整包装成架构改进。
- 把内部测试 seam 暴露成外部 Interface。
- 建议没有测试策略的重构。
- 忽略现有框架惯例带来的实际收益。

## 一句话原则

好的架构改进会把变化、知识和测试集中到更小的接口之后。
