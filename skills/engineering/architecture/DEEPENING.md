# 深化浅模块

如何在给定依赖关系的情况下，安全地深化一组浅模块。使用 [架构词汇表](LANGUAGE.md) 中的术语：**module**、**interface**、**seam**、**adapter**。

## 依赖分类

评估深化候选时，先对其依赖分类。类别决定了深化后的模块如何跨 seam 进行测试。

### 1. 进程内（In-process）

纯计算、内存状态、无 I/O。始终可深化——合并模块，直接通过新接口测试。不需要 adapter。

### 2. 本地可替换（Local-substitutable）

有本地测试替代品的依赖（如 PGLite 替代 Postgres、内存文件系统）。若替代品存在则可深化。深化后的模块在测试套件中使用替代品运行测试。seam 是内部的；模块的外部接口不暴露 port。

### 3. 远程但自有（Remote but owned，Ports & Adapters）

跨网络边界的自有服务（微服务、内部 API）。在 seam 处定义一个 **port**（接口）。深模块持有逻辑；传输层作为 **adapter** 注入。测试使用内存 adapter，生产使用 HTTP/gRPC/消息队列 adapter。

建议形式：_"在 seam 处定义 port，为生产实现 HTTP adapter、为测试实现内存 adapter，这样逻辑就能集中在一个深模块里，即使它跨网络部署。"_

### 4. 真正外部（True external，Mock）

你不控制的第三方服务（Stripe、Twilio 等）。深化后的模块将外部依赖作为注入的 port 接收；测试提供 mock adapter。

## Seam 纪律

- **一个 adapter 意味着假设的 seam，两个 adapter 才是真实的 seam。** 除非至少有两个 adapter 可以证明（通常是生产 + 测试），否则不要引入 port。单 adapter seam 只是多一层间接。
- **内部 seam 与外部 seam。** 深模块可以有内部 seam（私有于其实现，供自身测试使用），也有外部 seam（在其接口处）。不要因为测试用到了内部 seam 就把它暴露在接口上。

## 测试策略：替换，而不是叠加

- 浅模块上的旧单元测试，一旦深化后模块接口处的测试存在，就应删除——它们变成了冗余。
- 在深化后模块的接口处写新测试。**接口就是测试表面。**
- 测试断言的是通过接口可观察到的结果，而不是内部状态。
- 测试应该能在内部重构后继续通过——它们描述行为，而不是实现。如果实现改变时测试也必须跟着改，说明测试越过了接口。
