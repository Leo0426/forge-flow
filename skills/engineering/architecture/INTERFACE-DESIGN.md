# 接口设计

当用户想为选定的深化候选探索多种接口方案时，使用以下并行子智能体模式。基于 Ousterhout 的"设计两次"原则——你的第一个想法很可能不是最好的。

使用 [架构词汇表](LANGUAGE.md) 中的术语：**module**、**interface**、**seam**、**adapter**、**leverage**。

## 流程

### 1. 界定问题空间

在启动子智能体之前，向用户说明选定候选的问题空间：

- 任何新接口都需要满足的约束
- 它依赖的依赖项，以及它们属于哪个类别（见 [深化浅模块](DEEPENING.md)）
- 一个粗略的示意代码草图，用来让约束具体化——不是方案，只是让约束更直观

展示给用户后，立即进入第 2 步。用户在阅读的同时，子智能体也在并行工作。

### 2. 启动子智能体

使用 Agent 工具并行启动 3 个及以上子智能体。每个子智能体必须为深化后的模块生成**截然不同**的接口。

为每个子智能体准备独立的技术简报（文件路径、耦合细节、来自 [深化浅模块](DEEPENING.md) 的依赖类别、seam 后面有什么）。简报独立于第 1 步中面向用户的问题空间说明。给每个智能体一个不同的设计约束：

- 智能体 1："最小化接口——最多 1-3 个入口点。最大化每个入口点的 leverage。"
- 智能体 2："最大化灵活性——支持多种用例和扩展。"
- 智能体 3："针对最常见调用方优化——让默认场景变得简单。"
- 智能体 4（如适用）："围绕 ports & adapters 设计跨 seam 依赖。"

在简报中同时包含 [架构词汇表](LANGUAGE.md) 词汇和 CONTEXT.md 词汇，确保每个子智能体用一致的架构语言和项目领域语言命名。

每个子智能体的输出：

1. 接口定义（类型、方法、参数——加上不变量、调用顺序、错误模式）
2. 使用示例（展示调用方如何使用它）
3. seam 后隐藏了什么实现
4. 依赖策略和 adapters（见 [深化浅模块](DEEPENING.md)）
5. 权衡——leverage 高在哪里，薄在哪里

### 3. 呈现与对比

逐一呈现设计方案，让用户能充分理解每一个，然后用文字对比。从 **depth**（接口处的 leverage）、**locality**（变化集中在哪里）和 **seam 位置**三个维度来对比。

对比完成后，给出你自己的推荐：你认为哪个设计最强以及原因。如果不同设计的某些元素组合起来会更好，提出混合方案。要有明确立场——用户需要的是有力的判断，而不是一份菜单。
