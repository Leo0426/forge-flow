# 测试驱动开发

TDD 在 ForgeFlow 中不是“先写一堆测试”，而是用一个失败测试照亮下一小步实现。每一轮都应该让 MVP 更接近一条可验证的真实路径。

## 哲学

好测试通过公共 Interface 验证行为，而不是验证实现细节。代码可以完全改变；测试不应该。

好测试像规格说明：它描述系统做什么，以及调用者能观察到什么。它应该能经受内部重构，因为它不关心模块内部如何拆分。

坏测试与实现耦合：mock 内部协作者、测试私有方法、断言调用次数，或绕过公共 Interface 查询内部状态。重命名内部函数就失败的测试，通常测的是实现而不是行为。

示例见 [tests.md](tests.md)，mock 指南见 [mocking.md](mocking.md)。

## 红绿重构

```text
RED -> GREEN -> REFACTOR
```

- RED：写一个只覆盖一个行为的失败测试。
- GREEN：写最少代码让这个测试通过。
- REFACTOR：在所有测试通过后降低复杂度。

绝不要在 RED 时重构。先到 GREEN，再整理结构。

## 反模式：水平切片

不要先写所有测试，再写所有实现。这会把测试写成对想象中结构的锁定。

```text
错误：
RED:   test1, test2, test3, test4
GREEN: impl1, impl2, impl3, impl4

正确：
RED -> GREEN: test1 -> impl1
RED -> GREEN: test2 -> impl2
RED -> GREEN: test3 -> impl3
```

每个测试都回应上一轮实现学到的事实。

## 工作流

### 1. 计划垂直切片

写代码前确认：

- 要实现的用户行为或工程能力是什么。
- 公共 Interface 是什么，调用者需要知道哪些契约。
- 哪些行为最重要，哪些边界可以后置。
- 是否存在 Deep Module 机会：小接口、大实现、隐藏变化。
- 是否有相关 CONTEXT / ADR 约束。

只在高影响不确定性会改变实现时向用户提问。否则采用最小合理默认值并继续。

### 2. Tracer bullet

先写一条最窄端到端路径：

```text
RED:   第一个关键行为失败
GREEN: 最小实现打通路径
```

这条路径证明 Interface、依赖和测试入口都可工作。

### 3. 增量循环

对剩余行为逐个循环：

- 一次只写一个测试。
- 测试只描述可观察行为。
- 只使用公共 Interface。
- 只写足够通过当前测试的代码。
- 不添加猜测性功能。

优先顺序：

1. 核心成功路径。
2. 最重要边界条件。
3. 最可能出错的异常路径。
4. 回归风险最高的旧行为。

### 4. 重构

全部 GREEN 后，按 [refactoring.md](refactoring.md) 查找机会：

- 消除重复。
- 加深模块，把复杂度移到小 Interface 后面。
- 简化参数、隐藏实现数据结构。
- 隔离变化点和外部依赖。
- 保持同层抽象一致。
- 在自然适用处使用模式；模式必须服务于简化。

每个重构步骤后运行测试。

### 5. CONTEXT 同步

实现完成后检查是否发现了新知识。满足以下任一条件时，更新 `CONTEXT.md`：

- 实现过程中引入了 CONTEXT.md 中没有的领域术语。
- 澄清了某个 Interface 的前置条件、后置条件或不变量。
- 确认了某个模块边界的职责范围。

满足以下条件时，通过 `decision-forge` 创建 ADR：

- 做出了影响架构、接口或数据模型的决策，且存在真实 trade-off。
- 没有上下文会让未来读者困惑。

没有新发现时跳过此步骤。

## 每轮检查清单

```text
[ ] 测试描述行为，而不是实现
[ ] 测试只使用公共 Interface
[ ] 测试能经受内部重构
[ ] 当前实现是让测试通过的最小实现
[ ] 没有添加猜测性功能
[ ] GREEN 后才重构
[ ] 新发现的领域术语或接口契约已同步到 CONTEXT.md
```

## Mock 规则

只在系统边界 mock：

- 第三方 API。
- 时间、随机性、文件系统等非确定性依赖。
- 难以稳定运行的外部服务。

不要 mock 自己控制的内部模块。若测试必须 mock 大量内部协作者，优先怀疑 Interface 或 Module 形状不对。

## 输出期望

执行 TDD 时，持续报告：

- 当前 RED 测试验证什么行为。
- GREEN 做了什么最小实现。
- REFACTOR 降低了哪类复杂度。
- 最后运行了哪些测试。

## 一句话原则

一次一个行为，穿过公共 Interface，先让系统工作，再让结构变深。
