# Code Review

审查 diff，而不是评价代码风格。目标是找到：正确性 bug、值得简化的结构、与 CONTEXT / ADR / 方法论的偏离。

和 `improve-architecture` 的边界：它做全局架构分析；这个 skill 做 per-change 审查，范围是当前 diff。

## 审查前准备

先读取：

- `git diff` 或用户提供的 diff。
- `CONTEXT.md`：确认领域术语使用是否正确。
- `docs/adr/`：确认变更是否违反已有架构决策。

无法获取 diff 时停止并告知用户。

## 三类审查

### 1. 正确性

找会导致错误行为的问题：

- 逻辑错误：条件判断、边界值、分支遗漏。
- 错误处理：异常路径未覆盖、错误被静默吞掉、部分成功未处理。
- 并发：共享状态、竞态条件、未加锁的写操作。
- 资源：连接、文件、锁未关闭或未释放。
- 契约违反：调用方传入 CONTEXT.md / ADR 定义的不变量之外的值未被拒绝。
- 安全边界：用户输入、外部 API 响应未校验就使用。

### 2. 简化与复用

找值得改但不阻塞合并的结构问题：

- 重复逻辑：三处以上相似代码，Rule of Three 触发抽象信号。
- 穿透方法：方法只转发调用，没有封装任何复杂度。
- 参数过多（>3）：考虑参数对象或 Builder。
- if/else 特例爆炸：同类行为变体可用 Strategy 收敛。
- 测试测的是实现：mock 内部协作者、断言调用次数、依赖私有状态。

### 3. 方法论对齐

检查变更是否与项目约定一致：

- 领域术语：CONTEXT.md 中已定义的概念是否被正确使用，有无同义词替代或混用。
- ADR 遵循：变更是否绕过或违反 `docs/adr/` 中的已确认决策。
- 层级抽象：是否在同一模块中混用了不同抽象层级（领域逻辑和技术细节混在一起）。
- 外部依赖泄露：DB 查询、网络协议、第三方 SDK 是否泄露到核心逻辑层。

## 严重程度

| 等级 | 含义 | 处理 |
| --- | --- | --- |
| blocking | 正确性问题，合并前必须修复 | 必须处理 |
| suggestion | 简化或对齐问题，值得改但不阻塞 | 酌情处理 |

## 输出格式

```text
审查范围：<文件列表 或 diff 摘要>

正确性：
- [blocking] <文件:行号> <问题描述>

简化与复用：
- [suggestion] <文件:行号> <问题描述>

方法论对齐：
- [blocking/suggestion] <问题描述>

CONTEXT 同步：
- <如果发现新领域术语或接口契约，列出建议写入 CONTEXT.md 的内容；无则省略>

总结：
- blocking N 项，suggestion M 项
```

## 禁止行为

- 报告风格偏好（命名、格式、注释多少）。
- 把架构全局问题塞进这里——转给 `improve-architecture`。
- 无 diff 就开始审查。
- 把 suggestion 升级为 blocking。
- 忽略 CONTEXT / ADR 直接给建议。

## 一句话原则

找会出错的地方和值得简化的地方；风格留给 linter。
