# Diagnose Lite

用于快速调试问题的最小闭环流程。不要先猜代码哪里错，先构建可重复的失败信号。

## 核心原则

- 没有复现，不开始调试。
- 一次只验证一个假设。
- 假设必须可证伪。
- 修复必须可验证。
- 诊断信息必须最小、精准、可清理。

## 阶段 1：最小复现

目标：构建一个可重复的失败输入。

优先方式：

1. 自动化测试。
2. CLI / HTTP 请求。
3. 最小脚本，直接调用目标 Interface。
4. 手动步骤，只有在自动化暂时不可行时使用。

要求：

- 稳定复现，或记录明确复现率。
- 执行快，优先控制在 5 秒内。
- 有明确失败信号：报错、输出不对、状态不对、性能指标异常。
- 尽量穿过公共 Interface，而不是直接调用内部实现细节。

无法复现时停止继续猜测，向用户索要输入数据、请求参数、日志、trace、环境或复现步骤。

## 阶段 2：确认问题

检查：

- [ ] 复现的是用户描述的问题。
- [ ] 多次运行结果一致，或复现率已记录。
- [ ] 已记录具体错误信息、输出差异或 timing。
- [ ] 已确认当前分支和环境。

## 阶段 3：提出假设

生成约 3 个可验证假设，并按可能性和验证成本排序。

格式：

```text
如果 <原因>，那么 <改变某条件> 会导致 <结果变化>。
```

示例：

- 如果输入数据缺少必需字段，那么替换为完整输入后错误会消失。
- 如果状态污染导致失败，那么清理缓存或重启进程后复现率会变化。
- 如果并发导致竞态，那么单线程运行会稳定通过。

无法验证的假设应丢弃。

## 阶段 4：验证假设

一次只验证一个假设。

优先方式：

1. debugger / 断点。
2. 精准日志，使用前缀 `[D-LITE]`。
3. 临时 print。
4. profiler 或 timing，适合性能问题。

规则：

- 只添加必要信息。
- 禁止全量日志。
- 每次验证前说明期望结果。
- 验证失败后更新假设，不要硬拗原假设。

性能问题必须先测量，再优化。

## 阶段 5：修复和再验证

步骤：

1. 修改最小必要代码。
2. 运行最小复现，确认问题消失。
3. 运行原始场景，确认用户问题消失。
4. 运行相关回归测试，确认没有破坏已有行为。

如果修复改变了公共 Interface、模块边界或长期决策，交给 `decision-forge` 判断是否需要 CONTEXT 或 ADR。

## 阶段 6：清理和记录

- 删除 `[D-LITE]` 日志和临时 print。
- 删除或整理临时脚本。
- 保留能防止回归的测试。
- 简要记录根因、修复和验证命令。
- 如果根因揭示了 CONTEXT.md 中未记录的领域不变量或系统约束，更新 `CONTEXT.md`。

## 输出格式

```text
复现：
- ...

假设：
1. ...
2. ...
3. ...

验证：
- ...

根因：
- ...

修复：
- ...

验证结果：
- ...
```

## 禁止行为

- 没有复现就大规模读代码和猜测。
- 同时验证多个假设。
- 把临时日志留在最终代码中。
- 只修复症状，不确认根因。
- 性能问题不测量就优化。

## 一句话流程

复现 -> 假设 -> 验证 -> 修复 -> 再验证。
