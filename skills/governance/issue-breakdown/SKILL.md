# To Issues

把 PRD 或设计拆成可以领取、实现、验证的工程切片。切片必须端到端交付一条窄路径，而不是按 UI、API、数据库、测试做水平拆分。

## 核心原则

- 垂直切片优先：每个任务都能独立演示或验证。
- MVP 优先：先打通最短价值路径，再补边界和增强能力。
- AFK 优先：尽量把任务写到无需额外上下文即可执行。
- HITL 显式标注：需要人类判断、设计拍板或外部权限时，标为 HITL。
- 使用领域语言：标题和验收标准应对齐 `CONTEXT.md`，并尊重相关 ADR。

## 使用流程

### 1. 收集输入

读取当前对话、PRD、设计文档、已有 issue 或用户给出的计划。必要时探索代码库，确认：

- 现有 Module 和 Interface。
- 测试方式和可验证入口。
- 相关 flow、模板和 ADR。
- 哪些切片可以真正独立交付。

### 2. 切出 tracer bullets

每个切片都应包含：

- 一个可观察用户行为或工程能力。
- 穿过必要层次的最小实现路径。
- 明确验收标准。
- 测试或验证方式。
- 依赖关系。

避免水平任务：

```text
坏：先建 schema、再写 API、再写 UI、最后补测试
好：先交付一个最小用户路径，包含 schema/API/UI/tests 中必要的最小部分
```

### 3. 标注任务类型

| 类型 | 含义 |
| --- | --- |
| AFK | 信息完整，可由 agent 或开发者直接实现 |
| HITL | 需要人类判断、决策、权限、设计评审或外部协调 |

优先把任务写成 AFK。无法避免人类参与时，明确阻塞点。

### 4. 输出任务清单

先展示拟议拆分，等待确认或直接作为本地任务文档。

```markdown
## 任务拆分

1. <标题>
   类型：AFK / HITL
   被阻塞于：无 / <任务编号>
   覆盖用户故事：<编号或描述>
   要构建什么：<端到端行为>
   验收标准：
   - [ ] ...
   验证方式：
   - ...

## 推荐执行顺序

1. ...
```

### 5. 发布规则

- 默认输出为本地 markdown 任务清单。
- 如果仓库明确配置了 issue tracker，可以按依赖顺序发布。
- 发布外部 issue 时使用仓库已有 triage 标签；没有配置时不创建或猜测标签。
- 不关闭、不重写父 issue，除非用户明确要求。

## 切片质量检查

每个任务发布或交付前检查：

- 是否能独立验证？
- 是否避免了纯水平拆分？
- 是否有明确完成信号？
- 是否包含测试或手动验证方式？
- 是否只依赖前置切片，而不是依赖隐含上下文？
- 是否使用项目领域语言？

## 禁止行为

- 把 PRD 机械拆成“前端 / 后端 / 数据库 / 测试”。
- 创建无法验证完成的任务。
- 把需要决策的问题伪装成 AFK。
- 强制发布到外部 issue tracker。
- 在任务里塞入过多实现细节，导致后续重构成本变高。
