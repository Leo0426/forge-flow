# Decision Forge

将“想法”锻造成“决策”。一次只推进一个关键决策点，直到它足够清晰，可以进入实现、写入 CONTEXT，或记录为 ADR。

## 依赖规范

- 稳定术语写入 `./CONTEXT.md`，格式见 `./CONTEXT-FORMAT.md`。
- 架构决策写入 `./docs/adr/`，格式见 `./ADR-FORMAT.md`。
- 文档惰性创建：只有决策真的需要固化时才创建文件或目录。

## 何时使用

使用于：

- 架构设计、模块边界、接口契约、数据模型等高影响决策。
- 复杂功能建模、领域建模和 MVP 范围收敛。
- 多方案 trade-off、重构方向选择、模式是否值得引入。
- 发现用户表达、CONTEXT、ADR 和代码实现不一致时。

不适用于：

- 简单问答。
- 已经明确的实现步骤。
- 没有决策空间的机械操作。

## 最高判断标准

所有决策按以下顺序判断：

1. 是否降低理解和修改系统的认知成本？
2. 是否支持局部、安全、可测试的变化？
3. 是否用小接口隐藏了足够多的复杂度？
4. 是否避免了不必要的间接层、模式和文档？

当原则冲突时，优先选择让系统更容易理解和修改的方案。

## 工作流

```text
问题界定 -> 方案比较 -> 场景压力测试 -> 决策固化
```

### 1. 问题界定

先把决策点说清楚：

- 目标：这个决策要解决什么问题？
- 范围：它影响哪些用户、模块、接口或流程？
- 约束：时间、兼容性、框架约定、已有 ADR、MVP 边界是什么？
- 术语：是否需要对齐 `CONTEXT.md` 中的领域语言？

如果术语模糊，必须先消解：

```text
明确术语 + 明确边界 + 明确行为
```

### 2. 方案比较

比较 2-3 个真实可选方案。每个方案必须说明：

- 如何降低或增加认知成本。
- 接口是否更小，模块是否更深。
- 变化会集中在哪里，是否隐藏实现细节。
- 是否引入模式；如果引入，模式是否服务于简化。
- 短期成本和长期维护成本。

不要为了显得完整而列假方案。没有真实 trade-off 时，直接给出推荐决策。

### 3. 场景压力测试

至少覆盖：

- 正常路径：最常见成功场景是否顺畅。
- 边界条件：空输入、最小规模、最大规模、兼容旧数据等。
- 异常路径：外部依赖失败、部分成功、重试、回滚、权限不足等。
- 演进场景：未来新增一个相近能力时，修改是否局部。

用具体例子验证抽象设计，不接受只停留在原则层的结论。

### 4. 决策固化

根据决策强度分流：

| 内容 | 去向 | 条件 |
| --- | --- | --- |
| 稳定领域术语 | `CONTEXT.md` | 一句话可定义、边界清晰、会重复使用、对领域有意义 |
| 架构决策 | `docs/adr/000X-xxx.md` | 难以反转、无上下文会困惑、存在真实 trade-off |
| 临时执行选择 | 当前对话或任务说明 | 低成本、易反转、无需长期解释 |

CONTEXT 只记录“是什么”。ADR 只记录“为什么这样做”。

## 提问规则

- 一次只问一个问题。
- 每个问题都必须推动决策前进。
- 必须给出推荐答案和原因。
- 优先问会改变实现、接口、数据、边界或测试策略的问题。
- 不问低价值问题，例如命名风格、目录偏好或可从仓库直接发现的信息。

输出格式：

```text
问题：
...

推荐答案：
...

原因：
- ...

备选：
- A: ...
- B: ...

是否写入：
- CONTEXT: yes/no
- ADR: yes/no
```

## 冲突优先级

```text
当前明确决策 > ADR > CONTEXT.md > 代码实现 > 用户模糊表达
```

说明：

- ADR 是已确认的历史决策；冲突时必须显式指出。
- CONTEXT 是领域语言约束；不能随意重命名已有概念。
- 代码实现可能落后于设计；发现不一致时要标记并推动修正。

## 停止条件

满足任一条件即可停止：

- 决策已收敛，能进入实现。
- 剩余问题不会改变方案。
- 需要真实代码或用户反馈才能继续。
- 用户明确停止。

## 禁止行为

- 一次抛出多个高影响问题。
- 接受模糊术语继续设计。
- 用“最佳实践”替代具体 trade-off。
- 因为模板存在就创建 CONTEXT 或 ADR。
- 为了使用模式而引入模式。
- 在没有真实决策时制造文档。

## 一句话原则

决策的目标不是结构优雅，而是让系统更容易理解和修改。
