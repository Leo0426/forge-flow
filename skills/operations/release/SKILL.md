# Release

Release 用于把“代码完成”推进到“可以交付”。它关注风险、可回退性和发布后确认。

## 核心原则

- 可验证后再发布。
- 可回退后再发布。
- 变更范围必须清楚。
- 用户可见变化必须有说明。
- 发布后要确认真实环境健康。

## 使用流程

### 1. 收集发布内容

读取：

- 已完成任务或 issues。
- PRD、任务拆解和测试记录。
- code review、安全审计和诊断结果。
- 相关 ADR、迁移说明和兼容性约束。

### 2. 发布前检查

检查：

- 自动化测试是否通过。
- 手动 QA 是否覆盖核心路径。
- 是否有 breaking change。
- 是否需要数据迁移或配置变更。
- 是否影响安全、权限、支付、生产数据或外部依赖。

### 3. 生成发布材料

至少包含：

- Release summary。
- 用户可见变化。
- 风险和兼容性说明。
- Rollback plan。
- 发布步骤。
- 发布后验证步骤。

### 4. 发布后确认

确认：

- 核心路径可用。
- 错误率、性能和日志无异常。
- 用户反馈入口可用。
- 需要进入 `operations/operate` 的监控项已记录。

## 输出格式

```markdown
## 发布清单

范围：

发布前：
- [ ] 测试通过
- [ ] 手动 QA 完成
- [ ] 安全审查通过

发布说明：

风险：

回滚方案：

发布后验证：
- [ ] ...
```

## 禁止行为

- 没有 rollback plan 就发布高风险变更。
- 把测试通过等同于发布完成。
- 忽略用户可见变化说明。
- 不记录发布后验证结果。
