# 统一架构设计方法论

## 一、方法论框架

### 1. 三层结构

| 层级  | 来源                  | 解决问题   | 输出      |
| --- | ------------------- | ------ | ------- |
| 哲学层 | Ousterhout          | 为什么设计  | 复杂度控制目标 |
| 原则层 | SOLID / Clean Code  | 如何约束设计 | 设计规则    |
| 模式层 | GoF Design Patterns | 如何实现设计 | 结构模板    |

> 哲学层定义目标，原则层约束方向，模式层提供工具。模式服务于原则，原则服务于哲学。

### 2. 核心统一原则

| 原则ID | 原则       | 来源            | 本质         |
| ---- | -------- | ------------- | ---------- |
| U1   | 降低复杂度优先  | Ousterhout    | 控制认知成本     |
| U2   | 深模块优先    | Ousterhout    | 小接口 + 强实现  |
| U3   | 接口最小化    | ISP           | 降低使用成本     |
| U4   | 隐藏变化     | OCP + 信息隐藏   | 局部化变更影响    |
| U5   | 依赖抽象     | DIP           | 解耦实现       |
| U6   | 组合优于继承   | GoF           | 降低结构耦合     |
| U7   | 层级抽象一致   | Ousterhout    | 防止抽象泄露     |
| U8   | 模式服务于简化  | 经验法则          | 防止过度设计     |

### 3. SOLID 原则完整映射

| 原则  | 全称   | 核心约束             | 违反表现          | 修复方向              |
| --- | ---- | ---------------- | ------------- | ----------------- |
| SRP | 单一职责 | 一个类只有一个变化原因      | 类承担多个不相关能力    | 拆分、Strategy       |
| OCP | 开闭原则 | 扩展开放，修改关闭        | 新功能需修改旧代码     | Strategy、Decorator |
| LSP | 里氏替换 | 子类可替换父类且不破坏契约    | 子类重写方法改变语义    | 重新设计继承体系          |
| ISP | 接口隔离 | 客户不应依赖它不使用的接口    | 接口方法过多        | 拆分接口、Facade       |
| DIP | 依赖倒置 | 高层不依赖低层，都依赖抽象    | 直接依赖具体实现      | Adapter、DI        |

## 二、设计哲学基础

### 1. 战术编程 vs 战略编程

这是 Ousterhout 最核心的区分，是所有设计决策的出发点。

| 维度   | 战术编程      | 战略编程       |
| ---- | --------- | ---------- |
| 目标   | 让功能快速运行   | 让系统易于理解和修改 |
| 对复杂度 | 接受并绕过     | 主动识别并降低    |
| 时间成本 | 短期快，长期慢   | 短期慢，长期快    |
| 结果   | 补丁累积，腐化加速 | 结构演进，可持续维护 |

**规则**：每次设计决策，先问："这会降低还是增加系统的认知成本？"

### 2. 复杂度定义

> **复杂度 = 理解 + 修改系统所需的认知成本**

复杂度是累积的，每个设计决策都在增加或降低系统的长期成本。

### 3. 复杂度来源与方法论映射

| 复杂度来源      | 表现              | 对应原则              | 对应模式                       |
| ---------- | --------------- | ----------------- | -------------------------- |
| Change Amplification | 改一处动多处    | OCP               | Facade、Mediator            |
| Cognitive Load       | 需要记太多细节   | ISP + Deep Module | Facade                     |
| Unknown Unknowns     | 不知道要改哪里   | DIP + 信息隐藏    | Adapter                    |
| 特例爆炸       | if/else 泛滥      | SRP               | Strategy、Interpreter       |
| 穿透方法       | 方法仅转发调用，不封装    | Deep Module       | 重新设计层级边界                   |
| 时序耦合       | 调用顺序有隐式依赖       | 信息隐藏              | Builder                    |
| 层级抽象不一致    | 高低层逻辑混用在同一模块   | SRP + U7          | 按层拆分                       |
| 错误处理复杂度    | 错误类型爆炸，处理分散     | SRP               | Strategy                   |

## 三、模块设计规范

### 1. Deep Module 标准

| 维度    | Deep Module | Shallow Module |
| ----- | ----------- | -------------- |
| 接口复杂度 | 低           | 高              |
| 实现复杂度 | 高           | 低              |
| 封装能力  | 强           | 弱              |
| 可复用性  | 高           | 低              |

深模块的核心：**隐藏足够多的实现复杂度，使调用者不需要了解内部细节。**

### 2. 模块设计规则

| 规则        | 对应原则    | 说明                    |
| --------- | ------- | --------------------- |
| 模块必须封装复杂度 | U2      | 不向外暴露内部结构             |
| 模块职责必须完整  | SRP（增强） | 不是"单一函数"，而是"单一能力域"    |
| 模块接口必须稳定  | OCP     | 不因实现变化而变化             |
| 模块依赖必须抽象  | DIP     | 面向接口编程                |
| 同层保持同级抽象  | U7      | 禁止高低层逻辑混用于同一模块        |

### 3. 层级抽象一致性

> **不同层使用不同抽象级别；同一模块内，保持同级抽象。**

违反示例（领域逻辑与技术细节混用）：
```python
# 错误：领域层混入技术层
def process_order(order_id):
    conn = get_db_connection()           # 技术层
    sql = "SELECT * FROM orders WHERE …" # 技术层
    order = conn.execute(sql).fetchone()
    if order["status"] == "pending":     # 领域层
        ...
```

修复：领域层调用仓库接口，仓库实现处理技术细节。

### 4. 通用 vs 专用模块

| 维度    | 通用模块        | 专用模块        |
| ----- | ----------- | ----------- |
| 适用场景  | 核心稳定能力      | 特定业务场景      |
| 接口设计  | 宽泛但语义完整     | 精确但专用       |
| 复用性   | 高           | 低           |
| 变化频率  | 低           | 高           |

**规则**：核心能力倾向于通用模块；三次以上复用前不做通用抽象（Rule of Three）。

## 四、接口设计规范（Interface Contract）

### 1. 接口设计原则

| 原则    | 来源          | 要求             |
| ----- | ----------- | -------------- |
| 最小接口  | ISP         | 只暴露必要能力        |
| 语义完整  | Deep Module | 一次调用完成任务       |
| 不泄露实现 | 信息隐藏        | 不暴露内部数据结构和算法   |
| 稳定性   | OCP         | 不随实现变化而变化      |
| 语义一致性 | LSP         | 子类必须满足父类的完整契约  |

### 2. 接口契约（Contract）

接口不仅是方法签名，还包括：

| 契约要素  | 说明            |
| ----- | ------------- |
| 前置条件  | 调用方需满足的输入约束   |
| 后置条件  | 方法执行后的保证      |
| 不变量   | 对象生命周期内保持不变的属性 |

**LSP 本质**：子类实现可以替换父类，且不破坏任何已有契约——可以强化前置条件更宽松，后置条件更严格，但不能反向收紧。

### 3. 接口反模式

| 反模式      | 问题            | 修复方向       |
| -------- | ------------- | ---------- |
| 参数过多     | 高认知负担         | Builder / 参数对象 |
| 暴露内部数据结构 | 强耦合，阻碍重构      | 返回语义对象     |
| 多阶段调用    | 使用复杂，时序耦合     | 封装为单一调用    |
| 穿透方法     | 仅转发调用，不封装     | 重新设计层级边界   |
| 接口依赖具体实现 | 违反 DIP        | 提取抽象接口     |
| 子类破坏父类契约 | LSP 违反        | 重新设计继承关系   |

## 五、设计模式使用规范

### 1. 模式选择原则

| 原则           | 说明                     |
| ------------ | ---------------------- |
| 必须降低认知成本     | 模式引入后系统应更易理解，而非更难      |
| 不破坏 Deep Module | 不应将核心能力拆散到多个类        |
| 不引入不必要间接层    | 每个抽象层必须有明确价值           |
| 优先表达语义       | 用模式名字传达意图，而非炫耀结构       |

### 2. 模式分类与适用场景

**结构简化（降低 Cognitive Load）**

| 模式        | 解决问题           | 对应原则                 |
| --------- | -------------- | -------------------- |
| Facade    | 简化复杂子系统接口      | ISP + Deep Module    |
| Adapter   | 隔离外部依赖变化       | OCP + DIP            |
| Proxy     | 透明增加横切能力（缓存、日志）| OCP                  |
| Decorator | 可组合的行为扩展，不修改原类 | OCP + 组合优于继承         |

**行为变化（消除特例逻辑）**

| 模式              | 解决问题           | 对应原则      |
| --------------- | -------------- | --------- |
| Strategy        | 消除 if/else，替换算法 | SRP + OCP |
| Template Method | 固定流程骨架，替换步骤    | OCP       |
| Command         | 封装操作为对象，支持撤销   | SRP       |
| Observer        | 解耦事件源与处理者      | DIP       |
| Interpreter     | 表达规则/领域语言      | SRP       |

**对象创建（隐藏构造复杂度）**

| 模式               | 解决问题         | 对应原则      |
| ---------------- | ------------ | --------- |
| Builder          | 封装复杂对象构造     | 信息隐藏      |
| Factory Method   | 解耦对象创建与使用    | DIP       |
| Abstract Factory | 跨平台/多套实现切换   | OCP + DIP |

**流程组织（管理执行顺序）**

| 模式                     | 解决问题       | 对应原则      |
| ---------------------- | ---------- | --------- |
| Pipeline               | 分阶段处理，职责分离 | SRP       |
| Chain of Responsibility | 动态决定处理者    | OCP       |

### 3. 模式选择决策

```
问题属于哪类？
 接口太复杂，调用方负担高     → Facade
 依赖外部系统或需透明增强     → Adapter / Proxy
 行为有变体，if/else 泛滥   → Strategy / Template Method
 需要表达领域规则           → Interpreter
 对象构造步骤复杂           → Builder / Factory
 组件间事件/通知解耦         → Observer
 需扩展行为但不修改类         → Decorator / Proxy
 顺序处理，职责分离          → Pipeline / Chain
```

### 4. 模式滥用风险

| 模式         | 风险          | 判断标准              |
| ---------- | ----------- | ----------------- |
| Factory 过多 | 增加间接性，难以追踪  | 若无需解耦创建，直接实例化     |
| Visitor    | 泄露对象结构，破坏封装 | 考虑是否用多态替代         |
| Singleton  | 隐式全局依赖      | 优先用 DI 注入         |
| 过度继承       | 紧耦合，LSP 难保证 | 组合替代              |
| Mediator   | 退化为上帝对象     | 检查是否违反 SRP        |

## 六、架构分层规范

### 1. 标准分层模型

| 层级             | 职责     | 原则                | 常用模式              | 变化频率 |
| -------------- | ------ | ----------------- | ----------------- | ---- |
| Interface      | 对外接口   | ISP               | Facade            | 中    |
| Application    | 流程编排   | SRP               | Pipeline、Command  | 高    |
| Domain/Core    | 核心业务能力 | Deep Module + SRP | Strategy、Observer | 低    |
| Infrastructure | 技术实现   | DIP               | Adapter、Proxy     | 高    |

### 2. 分层约束

| 规则                        | 说明               |
| ------------------------- | ---------------- |
| 上层依赖下层抽象                  | 不依赖具体实现          |
| 禁止跨层调用                    | 防止耦合蔓延           |
| Domain 层最稳定               | 不引入技术细节          |
| Infrastructure 不泄露到 Domain | 技术概念不进入业务层       |
| 同层保持同级抽象                  | 避免层级抽象不一致（U7）    |

## 七、变化管理规范（Evolution Control）

### 1. 变化类型与隔离

| 变化类型    | 隔离层             | 原则  | 模式       |
| ------- | --------------- | --- | -------- |
| 业务规则变化  | Domain 层        | SRP | Strategy |
| 技术实现变化  | Infrastructure 层 | DIP | Adapter  |
| 接口协议变化  | Interface 层     | ISP | Facade   |
| 跨系统事件变化 | Application 层   | OCP | Observer |

### 2. 演进规则

| 规则        | 说明                     |
| --------- | ---------------------- |
| 新功能不修改旧代码 | OCP —— 通过扩展实现          |
| 优先扩展而非修改  | 减少 Change Amplification |
| 局部修改优先    | Deep Module —— 修改影响范围最小 |
| 重构与功能分离提交 | 防止重构引入业务 bug            |

## 八、原则冲突与权衡

实际设计中原则之间可能冲突，需要明确权衡准则。

| 冲突场景          | 原则 A  | 原则 B  | 权衡准则                               |
| ------------- | ----- | ----- | ---------------------------------- |
| 深模块 vs SRP   | 高封装   | 单一职责  | 优先深模块；SRP 以"变化原因"而非"行数"判断         |
| 通用性 vs 简单性   | 可复用   | 避免复杂度 | 三次使用前不抽象；核心能力才通用                   |
| 模式语义 vs 间接性  | 语义表达  | 降低间接  | 只在语义收益 > 间接成本时引入模式                 |
| OCP vs 实用性   | 扩展不修改 | 工程成本  | 对高频变化点做扩展设计；低频点直接修改               |
| LSP vs 灵活性   | 契约一致  | 子类定制  | LSP 不可破坏；通过组合实现定制，不依赖违反契约的继承      |

**总权衡准则**：当原则冲突时，以"是否降低系统的认知成本"为最终判断标准（U1 优先）。

## 九、复杂度控制机制（落地）

### 1. 设计评审检查表

| 检查项          | 判断标准             |
| ------------ | ---------------- |
| 接口是否简单       | 参数少，语义清晰         |
| 模块是否 Deep    | 接口简单，实现封装完整      |
| 是否泄露实现       | 是否暴露内部数据结构或算法    |
| 是否存在穿透方法     | 方法是否仅转发调用而不封装    |
| 是否层级一致       | 同一模块是否混用不同抽象层    |
| 是否存在特例爆炸     | if/else 是否应替换为策略 |
| 是否可局部修改      | 修改影响范围是否可控       |
| 是否存在时序耦合     | 是否有隐性调用顺序依赖      |
| 子类是否满足父类契约   | 是否符合 LSP         |
| 依赖是否面向抽象     | 是否直接依赖具体实现       |

### 2. 设计警示信号（Smells）

| 警示信号           | 对应问题                  | 处理方向              |
| -------------- | --------------------- | ----------------- |
| 方法参数过多         | 认知负担高                 | Builder / 参数对象    |
| 类承担多个不相关能力     | SRP 违反                | 拆分职责              |
| if/else 嵌套过深   | 特例爆炸                  | Strategy          |
| 修改一处引发多处改动     | Change Amplification  | OCP + Facade      |
| 直接 new 具体类     | DIP 违反                | DI / Factory      |
| 方法仅转发调用        | 穿透方法，层级设计不当           | 重新设计边界            |
| 子类重写方法改变语义     | LSP 违反                | 重新设计继承            |
| 单元测试需要大量 Mock  | 依赖太具体                 | DIP + Adapter     |
| 同层方法抽象级别差异大    | 层级抽象不一致               | 按层拆分              |

### 3. 可量化指标（参考基准）

> 这些数字是参考信号，不是硬规则。复杂度降低是目标，指标是辅助判断手段。

| 指标        | 参考上限 | 超出时考虑           |
| --------- | ---- | --------------- |
| 函数参数数量    | 3    | Builder / 参数对象  |
| 接口方法数量    | 5    | ISP 拆分          |
| if/else 嵌套深度 | 2 | Strategy        |
| 模块直接依赖数   | 5    | 检查 DIP          |
| 函数行数      | 30   | 检查 SRP          |

## 十、核心设计结论

### 方法论统一

| 方法              | 作用    | 核心问题            |
| --------------- | ----- | --------------- |
| Ousterhout      | 控制复杂度 | 系统是否容易理解和修改？    |
| SOLID           | 约束结构  | 设计是否支持安全演进？     |
| Design Patterns | 提供实现  | 如何用成熟结构表达设计意图？  |

### 设计决策优先级

1. **是否降低认知成本？**（Ousterhout / 哲学层）
2. **是否支持安全变化？**（SOLID / 原则层）
3. **是否有成熟模式可用？**（GoF / 模式层）

### 最终原则

> **设计的目标不是"结构优雅"，而是"系统容易理解和修改"。**

> **用最少的接口，封装最多的复杂度；选择合适的模式，实现最简单的系统。**
