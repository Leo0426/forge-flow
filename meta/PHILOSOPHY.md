# ForgeFlow 核心思路
把优秀工程师的工作方法和工程行为协议化。而不是让AI自由发挥写代码。

## 定位与问题

| 维度 | 思路 |
| --- | --- |
| 核心定位 | 面向真实工程开发的 Claude/Codex 技能集合，不是"氛围编程" |
| 解决问题 | AI 误解需求、输出啰嗦、代码不可运行、架构变烂 |
| 设计原则 | 小而独立、可组合、易修改、跨模型可用 |
| 关键机制 | 用固定流程约束Agent行为，例如提问澄清、TDD、诊断、架构复盘 |
| 项目记忆 | 通过CONTEXT.md维护领域语言，通过docs/adr维护架构决策 |
| 工程闭环 | 需求澄清 → PRD → Issue 拆分 → TDD 实现 → Diagnose 调试 → 架构改进 |

## 关键 Skills

每个 Skill 对应一种工程师思维模式，把隐式的专家判断显式化为 Agent 可以执行的流程。

| Skill | 作用 | 背后方法论 |
| --- | --- | --- |
| `/decision-forge` | 让 Agent 反复追问需求，避免一上来就写代码 | 需求澄清 |
| `/grill-with-docs` | 追问需求，同时沉淀领域语言和 ADR | DDD + 架构决策记录 |
| `/tdd` | 要求 Agent 按 red-green-refactor 写代码 | 测试驱动开发 |
| `/diagnose` | 用"复现 → 最小化 → 假设 → 插桩 → 修复 → 回归测试"调试 | 系统化 Debug |
| `/issue-breakdown` | 把计划拆成可独立执行的 issue | 垂直切片 |
| `/architecture-audit` | 定期让 Agent 从系统角度找架构恶化点 | 架构治理 |
| `/zoom-out` | 让 Agent 不只看局部代码，而是解释系统上下文 | 上下文补全 |

## 核心洞察

> **把"工程师怎么思考"显式流程化，再交给 Agent 执行。**

AI 的根本问题不是能力不足，而是缺乏约束——它不知道什么时候该停下来问、什么时候该写测试、什么时候该从全局视角看问题。ForgeFlow 的每个 Skill 就是一个约束单元，把这些判断时机固化成可触发的流程。

这意味着：
- Agent 不是在"自由发挥"，而是在执行一个被设计过的工程规程
- 人的判断没有被替代，而是被放在了关键决策节点上
- 工程方法可以随着项目积累而沉淀，而不是每次从零开始

## 对 AIGC + 静态分析智能体的启发

类似的 Skill 化设计可以直接迁移到其他工程域。以告警研判 + 静态分析场景为例：

| 场景 | 可借鉴的 Skill 化设计 |
| --- | --- |
| 告警研判 | `/triage-alert`：按资产、漏洞、攻击链、历史误报分层研判 |
| 误报分析 | `/diagnose-false-positive`：复现规则命中路径，定位误报原因 |
| 规则生成 | `/to-rule-spec`：从漏洞描述生成规则规格，而不是直接生成规则代码 |
| 规则测试 | `/tdd-rule`：先生成正/反例样本，再实现规则 |
| 上下文补全 | `/zoom-out-codebase`：分析调用链、数据流、框架语义 |
| 架构治理 | `/improve-analysis-pipeline`：定期检查规则爆炸、重复规则、上下文缺失 |

关键是**不要只做一个大而全 Agent**，而是按照工程师实际的思维步骤，把能力拆成独立、可组合的 Skill 单元。每个 Skill 解决一个具体问题，可以单独触发，也可以串联成 Flow。
